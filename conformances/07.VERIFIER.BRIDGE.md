# AAP Conformance Verifier Bridge

This artifact defines canonical object-form conformance cases for
`inconclusiveResolution=requireVerifierBridge`.

Retry accounting rule:

- `maxBridgeRetries` counts retries after the initial accepted submission
- `maxBridgeRetries: 1` therefore permits at most `2` total bridge attempts for the same
  `BridgeSubmission` binding tuple
- every retry must preserve `subjectDigest`, `evaluationScopeDigest`,
  `predicateRuntimeProfileDigest`, `proofProfileDigest`, and `contractRevision`
- a retry that changes any required binding is not a retry; it is a new inadmissible submission

## Timeout Case

### Precondition

- accepted bridge profile is `sha256:bridge_profile_gamma_0001`
- no bridge result arrives before `maxBridgeResponseMs=20000`

Expected outcome:

```text

  accepted: false
  bridgeAttemptsUsed: 1
  failureReason: bridgeTimeout
  mappedOutcome: failClosed
  contractMayProceedToCompletion: false
}
```

## Retry-Success Case

### Precondition

- the first accepted `BridgeSubmission` attempt timed out without a result
- one retry is emitted with the same binding tuple:
  `subjectDigest`, `evaluationScopeDigest`, `predicateRuntimeProfileDigest`,
  `proofProfileDigest`, and `contractRevision`
- the second attempt receives a passing bridge result before `maxBridgeResponseMs=20000`

Expected outcome:

```text
BridgeOutcome {
  accepted: true
  bridgeAttemptsUsed: 2
  terminalTrustLevel: verified
  mappedOutcome: bridgeReplayAccepted
  contractMayProceedToCompletion: true
}
```

## Retry-Exhaustion Case

### Precondition

- the initial accepted `BridgeSubmission` attempt timed out
- one retry is emitted with the same accepted binding tuple
- the retry also times out before `maxBridgeResponseMs=20000`

Expected outcome:

```text
BridgeOutcome {
  accepted: false
  bridgeAttemptsUsed: 2
  failureReason: bridgeTimeoutRetryExhausted
```

## Required Rule

`requireVerifierBridge` is portable only if:

- the contract names a bridge profile
- the bridge profile names the bridge domain and response bounds
- the active trust policy authorizes that bridge domain
- the bridge verifier proves authorized bridge membership in that domain
- the bridge submission and bridge result cover the same subject, evaluation scope, runtime,
  proof profile, and contract revision
- the evidence disclosed to the bridge stays within the accepted privacy class
- `maxBridgeRetries` counts retries after the initial attempt, not total attempts
- timeout, rejection, and policy block map to machine-determined outcomes
