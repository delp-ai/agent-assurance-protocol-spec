# AAP Wire Format Specification

## Purpose

This document defines the low-level binary framing model for `AAP (Agent Assurance Protocol)`.
The goal is to give agent nodes a compact, machine-first envelope for exchanging
contracts, evidence, state transitions, and results over a datagram-oriented fabric.

## Design Goals

- Minimize parsing ambiguity.
- Support multiple concurrent streams inside one secure session.
- Distinguish control traffic from bulk delivery traffic.
- Allow selective reliability by stream and by frame class.
- Preserve causal ordering without forcing total ordering.
- Make replay, audit, and verification practical.

## Layer Placement

AAP assumes the following order:

1. Datagram fabric
2. Secure session
3. Multiplexed streams
4. Typed frames
5. Contract and proof semantics

The AAP frame format begins only after a lower-layer session bootstrap has
produced authenticated peers and symmetric session keys.

Bootstrap packets used only for peer discovery, cryptographic setup, and anti-replay
window establishment are outside this document. `HELLO` and `CAPS` are the first
protected AAP control frames sent after bootstrap succeeds.

## AAP Base Profile

The autonomous-only `AAP Base Profile` is the minimum interoperable subset for
third-party agents.

Required properties:

- a lower-layer bootstrap transcript that yields authenticated peers, a replay window,
  and an encrypted session before the first AAP frame is processed
- protected `HELLO` exchange on stream `0` before any contract frame is accepted
- protected `CAPS` exchange on stream `1` before any optional extension is used
- portable retrieval or inline carriage of any identity, capability, schema, policy,
  or unit manifest referenced during handshake
- one portable freshness model for handshake manifests, revocation objects, and
  contract-visible validity windows
- canonical payload encoding for all base-profile frame bodies
- explicit wire receipt acknowledgment, duplicate detection, and replay-window handling
- deterministic handling for `CANCEL`, `SETTLE`, `CLOSE`, and wire errors
- no hidden dependence on local operator judgment or implementation-specific defaults

### Public Secure-Session Companion Profile

For controlled federation, the secure session below AAP may be pinned by deployment.

For unknown-peer public interoperability, that lower layer is itself part of the
protocol surface and must be standardized tightly enough that two independent agents can
derive the same authenticated transcript state before `HELLO`.

The public companion profile must define at minimum:

- transcript-hash scope for bootstrap packets, accepted handshake manifests, and negotiated
  session parameters
- authenticated byte coverage for `HELLO`, `CAPS`, and resumed bootstrap traffic
- derivation rules for `sessionLineageId`, `bootstrapTranscriptDigest`, and
  `resumeTokenBindingDigest`
- replay-window creation, persistence, restoration, and safe-refusal behavior
- resume-token format, freshness, peer binding, and negotiated-version-family binding
- deterministic failure reasons for rejected resumption, transcript divergence, or replay-state
  restoration failure

Companion profile registry:

- `1`: `aap-secure-session-v1`

Base-profile rule:

- an implementation may claim controlled-federation AAP conformance while pinning this profile
  out of band
- an implementation may claim unknown-peer public `AAP Open Core` interoperability only if it
  implements this companion profile together with the handshake, bootstrap, and safe-resume
  conformance artifacts published in `conformances/`

Companion-profile interoperability boundary:

- any implementation that delegates bootstrap or replay restoration below its AAP codec must
  still surface the resulting accepted state through canonical audit objects
- these objects are the machine-readable boundary between the lower secure session and AAP
  contract traffic; they prevent hidden local state from deciding whether resumed traffic is
  admissible

```text
BootstrapAcceptance {
  sessionLineageId
  bootstrapTranscriptDigest
  authenticatedPeerIds[]
  negotiatedProtectionProfile
  acceptedTrustPolicyDigest
  degradedBootstrapMode
  replayWindowDigest
  replayWindowCreatedAtUnixMs
  canonicalDigest
  signatureSet[]
}

ResumeAcceptance {
  priorSessionLineageId
  newSessionId
  bootstrapTranscriptDigest
  resumeTokenDigest
  resumeTokenBindingDigest
  acceptedTrustPolicyDigest
  degradedBootstrapMode
  restoredReplayWindowDigest
  restoredReliableMessageFloor[]
  acceptedAtUnixMs
  canonicalDigest
  signatureSet[]
}

ResumeRefusal {
  priorSessionLineageId
  attemptedBootstrapTranscriptDigest
  resumeTokenBindingDigest
  attemptedTrustPolicyDigest
  refusalReasonCode
  resultingAction
  refusedAtUnixMs
  canonicalDigest
  signatureSet[]
}
```

Boundary rules:

- a new `sessionLineageId` becomes contract-usable only after `BootstrapAcceptance` exists
- resumed contract traffic is admissible only if a `ResumeAcceptance` exists for the new
  transport `sessionId`
- `ResumeAcceptance.newSessionId` must carry that raw 16-byte transport `sessionId`, not a local
  textual alias, and it must byte-match the resumed frame-header `SessionId`
- `acceptedTrustPolicyDigest` is the current policy root for every later trust, bridge, and
  finality decision in the same session lineage
- `degradedBootstrapMode` is the authoritative session-lineage bootstrap outcome and must be one
  of `normal`, `diagnosticOnly`, `domainLimited`, or `finalityBlocked`
- unknown-peer public bootstrap is valid only if both peers converge on the same
  `acceptedTrustPolicyDigest`; local policy narrowing that changes the digest requires a fresh
  session family and fresh contract negotiation rather than silent reuse of the prior lineage
- if resumption is refused, the refusal must be auditable through `ResumeRefusal`; silent
  widening of the replay window is invalid
- `ResumeRefusal` is the canonical audit artifact for refusal, but the base profile does not carry
  it directly on wire; the peer-visible refusal signal is the terminal `WIRE_ERROR` required by the
  recovery rules below unless a later published extension standardizes explicit carriage
- the canonical digests of these objects must be stable across independent implementations even
  if the lower-layer secure-session profile is pinned locally

Companion-profile v1 refusal registry:

- `1`: `resumeTokenInvalid`
- `2`: `resumeTokenExpired`
- `3`: `peerBindingMismatch`
- `4`: `versionFamilyMismatch`
- `5`: `bootstrapTranscriptMismatch`
- `6`: `replayContinuityNotRestorable`
- `7`: `trustPolicyMismatch`
- `8`: `requiredAuditObjectUnavailable`

Companion-profile v1 resulting-action registry:

- `1`: `requireFreshSession`
- `2`: `closeTransportSession`
- `3`: `allowDiagnosticsOnly`

Stale-status outcome registry:

- `1`: `rejectEvidence`

Companion-profile v1 acceptance algorithm:

1. validate the lower-layer authenticated peers and bootstrap transcript bytes
2. derive `bootstrapTranscriptDigest` over the companion profile's canonical authenticated
   transcript scope
3. resolve and validate the active unknown-peer trust policy before `HELLO` or resumed
   contract traffic is accepted
4. create or restore the replay window and dedupe floor for reliable frames
5. if the session is fresh, emit `BootstrapAcceptance`; if it is resumed, emit
   `ResumeAcceptance`; otherwise emit `ResumeRefusal`
6. block contract-bound traffic until protected `HELLO` and `CAPS` complete under the accepted
   lineage and policy digest

### Base Profile Handshake

The session bootstrap below this layer authenticates peers and installs session keys.
After that point the first AAP interaction is:

1. initiator sends `HELLO`
2. responder sends `HELLO`
3. both sides exchange `CAPS`
4. both sides derive the best common `AAP` version and extension set
5. contract traffic may begin

If any step fails, the session must be closed without accepting contract-bound frames.

Bootstrap-version rule:

- `HELLO` and the pre-negotiation `CAPS` exchange must use fixed wire `Version=0.0`
- after both peers emit matching negotiated `chosenMajor` and `chosenMinor`, all subsequent
  AAP frames in that session must carry the negotiated wire version
- a resumed session repeats the same bootstrap-version rule before resumed contract traffic is
  accepted
- any resumed session that cannot restore replay continuity and prior transcript binding must
  fail resumption and start a fresh lineage instead of widening replay acceptance

### Portable Discovery Rule

The base profile must be joinable by an unknown third-party agent without vendor-local
configuration.

Therefore:

- any digest announced in `HELLO` or `CAPS` must be accompanied by a content-addressed
  manifest reference or an inline manifest body
- a peer may treat a pre-pinned manifest as already resolved, but must still verify that
  the announced digest matches the pinned artifact
- a peer must reject base-profile handshake success if mandatory discovery materials are
  neither resolvable nor pre-pinned by active policy
- manifest retrieval must be protected against substitution by digest binding and covered
  by the session transcript once accepted

## Portable Time And Freshness Model

AAP distinguishes local causal ordering from cross-domain freshness.

- local actor ordering may use logical counters or hybrid logical clocks
- any value that must be compared across independent cells must use absolute UTC time
- the base profile encodes absolute time as `unixMs`, an unsigned 64-bit millisecond
  count since the Unix epoch

Cross-domain validity checks must use a shared skew rule.

Base-profile rule:

- each peer evaluates absolute validity with `maxClockSkewMs`
- an object is not yet valid only if `localNowUnixMs + maxClockSkewMs < notBeforeUnixMs`
- an object is expired only if `localNowUnixMs - maxClockSkewMs > notAfterUnixMs`
- optional logical or hybrid-clock fields may assist audit ordering but must never extend
  or shorten portable validity windows
- if local time confidence is below policy minimum, the peer must refuse any operation
  that requires portable freshness rather than substituting a hidden local heuristic

This time model applies to:

- handshake manifests
- trust-domain attestations and revocation objects
- contract deadlines and success validity
- capability-token validity windows
- metering and settlement receipts

### Time And Revocation Handshake Profiles

`CAPS` negotiation is only portable when the named time and revocation profiles are themselves
machine-readable artifacts rather than undocumented local presets.

```text
TimeProfileManifest {
  profileId
  maxClockSkewMs
  absoluteTimeRequired
  notBeforeFieldRequired
  notAfterFieldRequired
  canonicalDigest
  signatureSet[]
}

RevocationProfileManifest {
  profileId
  statusObjectProfileId
  maxStatusAgeMs
  requireOnlineStatusForHighRisk
  staleStatusOutcome
  canonicalDigest
  signatureSet[]
}
```

Rules:

- `CapsBody.timeProfileId` must equal `TimeProfileManifest.profileId`
- `CapsBody.revocationProfileId` must equal `RevocationProfileManifest.profileId`
- `timeProfileRef` and `revocationProfileRef` must resolve to these canonical manifest objects when
  the profiles are not already policy-pinned
- independent implementations must evaluate freshness and stale-status outcomes from the resolved
  profile objects rather than from deployment-local defaults

## Manifest Discovery Profile

The base profile standardizes manifest carriage tightly enough for an unknown external
agent to join without vendor-local bootstrap logic.

### ManifestRef

```text
ManifestRef {
  manifestDigest
  digestAlgorithm
  mediaType
  retrievalProfile
  locatorSet[]
  inlineAllowed
  maxBytes
  notAfterUnixMs
}
```

Rules:

- `manifestDigest` is the content address of the canonical manifest bytes
- `retrievalProfile` names the retrieval contract, not an implementation-local plugin
- `locatorSet[]` is an ordered list of dereference locations or registry paths
- `maxBytes` is a sender assertion; receivers must still enforce their own policy ceiling
- `notAfterUnixMs` limits freshness of mutable references such as revocation manifests

### Base-Profile Discovery Rules

- every base-profile peer must support inline manifest carriage for handshake-critical
  manifests
- the base-profile `retrievalProfile` value for handshake-critical manifests is
  `inline-handshake-v1`
- `HELLO` and `CAPS` may rely on pre-pinned manifests, but otherwise their mandatory
  references must resolve from inline bodies carried in the same protected exchange
- a contract or proof frame that introduces a new mandatory manifest after `CAPS`
  must either carry it inline or reference a previously accepted manifest registry entry
- post-`CAPS` retrieval may use `session-registry-v1`, which means lookup through a
  previously accepted protected manifest registry within the same session
- external retrieval profiles may be negotiated, but they must not be required for
  base-profile join success
- manifest dependency chains must be acyclic and no deeper than `4` references in the
  base profile
- a peer must reject a manifest graph whose cumulative inline size exceeds `262144`
  bytes or whose individual manifest exceeds `65536` bytes unless a negotiated profile
  explicitly raises the limit
- a peer must reject unresolved mandatory manifests, cyclic references, digest mismatch,
  media-type mismatch, stale manifests, and policy-blocked manifests deterministically

### `session-registry-v1`

`session-registry-v1` is the only base-profile post-`CAPS` retrieval mode. It is a protected
in-session registry, not an implementation-local cache lookup.

```text
SessionRegistrySnapshot {
  registrySessionId
  registryRevision
  issuerAgentId
  createdAtUnixMs
  entries[]
  canonicalDigest
  signature
}

SessionRegistryEntry {
  manifestDigest
  mediaType
  manifestBytes
  firstAcceptedMessageRef
  expiresAtUnixMs
}
```

Rules:

- every `SessionRegistryEntry.manifestDigest` must equal the digest of `manifestBytes`
- `firstAcceptedMessageRef` must identify the accepted frame that first introduced the
  manifest into the protected session transcript
- a receiver must reject any lookup result whose `registrySessionId` does not match the
  current session lineage or accepted resumption lineage
- later `registryRevision` values may append entries, but must never mutate previously
  accepted manifest bytes for the same digest
- `expiresAtUnixMs` must be less than or equal to the manifest's own validity horizon when
  one exists
- if a peer resumes a session without restoring the accepted registry lineage safely, it
  must refuse `session-registry-v1` lookups and require fresh inline carriage instead

### Handshake Manifest Set

For unknown-peer interoperability, the protected `HELLO` and `CAPS` exchange must make
the following objects available by inline body or verified pre-pin:

- identity manifest
- current agent-identity attestation or a canonical manifest that embeds it
- capability-set manifest
- schema-registry manifest
- unit-registry manifest
- time profile manifest
- revocation profile manifest
- active `TrustPolicyManifest`
- current trust-domain attestation and admission-basis object whenever the peer claims a
  trust domain that may later contribute to verifier independence, quorum, or witnessed finality

Handshake completion rule:

- `CAPS` is not complete until the peer has validated the advertised time and revocation
  profiles and determined whether the chosen profile requires freshness or transparency
  checks before session activation
- if the negotiated revocation profile requires current status for peer identity or trust
  bootstrap artifacts, those checks must succeed before contract traffic begins
- if the peers advertise different trust policy digests in bootstrap or `HELLO`, session setup
  must fail or enter `diagnosticOnly` degraded mode; contract traffic must not begin under hidden
  policy asymmetry
- if the negotiated finality or transparency profile is unknown, the peer may keep the
  session alive only in `diagnosticOnly` degraded mode and must reject contract-bound frames
- if the peer's identity attestation is current but its domain attestation or admission-basis
  object is missing, stale, or policy-rejected, the peer may remain authenticated for traffic
  that does not require domain independence, but it must enter `domainLimited` degraded mode and
  must not be treated as quorum-eligible for trust or finality

Degraded bootstrap modes:

- degraded handling is allowed only after peers have authenticated the protected session and
  converged on one `chosenMajor`, `chosenMinor`, `chosenProfiles[]`, and trust-policy result
- `diagnosticOnly` permits only `HELLO`, `CAPS`, `WIRE_ERROR`, and session-scope `CLOSE`;
  contract-bound frames and semantic state mutation are invalid
- `domainLimited` permits contract traffic only when the negotiated profile allows progress at
  `authenticated` without domain-independence claims; any frame or object that would claim
  `quorumVerified`, `witnessQuorum`, bridge-domain eligibility, or quorum-gated finality is
  invalid until the session exits degraded mode
- `finalityBlocked` permits non-terminal contract traffic only when the negotiated profile allows
  progress below the finality floor, but `SETTLE` that claims irreversible settlement and contract
  `CLOSE` are invalid until finality prerequisites are satisfied
- a deployment may refuse any degraded mode and fail the session instead, but it must not silently
  widen degraded permissions beyond the profile above
- exiting degraded mode requires the missing bootstrap prerequisite to become valid and both peers
  to continue using the already-negotiated version, profile, and trust-policy binding; otherwise
  the session must close and restart

### Degraded Bootstrap Admission Profile

| Key                                  | Value                                                                                                                       |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------- |
| `domainLimitedAllowedPhases`         | `accepted, ready, executing, challenged, delivering, proved, completed, partiallyCompleted, failed, rollingBack, cancelled` |
| `domainLimitedRejectedTrustLevels`   | `quorumVerified`                                                                                                            |
| `domainLimitedRejectedFinalityModes` | `witnessQuorum, transparencyAnchor`                                                                                         |
| `domainLimitedAllowsBridgeDomains`   | `false`                                                                                                                     |
| `domainLimitedAllowsQuorumClaims`    | `false`                                                                                                                     |
| `finalityBlockedAllowedPhases`       | `accepted, ready, executing, challenged, delivering, proved, completed, partiallyCompleted, failed, rollingBack, cancelled` |
| `finalityBlockedForbiddenFrames`     | `SETTLE, CLOSE`                                                                                                             |
| `finalityBlockedLateEvidencePolicy`  | `retainForAuditOnly`                                                                                                        |

Determinism rules:

- `domainLimited` is transport-authenticated but not domain-eligible; any artifact that would grant
  `quorumVerified`, admit a bridge domain, or claim quorum-gated finality is invalid until domain
  eligibility is restored
- `finalityBlocked` may still admit result-freezing `STATE` transitions through the allowed phases
  above, but it must reject `SETTLE` and contract `CLOSE` until finality prerequisites are
  satisfied or the accepted contract maps the blockage through `blockedCloseOutcome`
- late finality, witness, transparency, or revocation evidence that arrives after a
  `finalityBlocked` terminal resolution must follow `retainForAuditOnly`

Once accepted, the canonical digests of all handshake manifests become part of the
bootstrap transcript binding and must be surfaced to audit and replay.

## Byte Order And Encoding Rules

- All fixed-width integers are unsigned big-endian.
- Variable-length integers use `uvarint` encoding.
- Standalone binary envelopes are length-prefixed; payload field bytes rely on their enclosing
  field length.
- Text fields are UTF-8 only when text is unavoidable.
- Identifiers that participate in canonical negotiation ordering must be restricted to visible
  ASCII and compared by raw bytewise lexicographic order.
- Enumerations are numeric on the wire.
- Optional fields are encoded by omission unless a schema explicitly marks presence as
  semantically significant.
- Unknown tagged fields must be ignored unless marked critical.
- `TTL` and all wire-local durations are encoded in milliseconds only.
- Base-profile structured payloads use the fixed canonical encoding profile
  `aap-tagbin-v1`.
- Payload digests and audit digests for `HELLO`, `CAPS`, and all other base-profile
  frames are computed over `aap-tagbin-v1` bytes.
- Negotiated alternate payload encodings may be introduced only by a post-`CAPS`
  extension that preserves canonical digest stability for any artifact already bound by
  the accepted contract or trust state.
- signed or digested base-profile objects that are not frame payloads must use the canonical
  field assignments defined in `04.CANONICAL.OBJECTS.md`

Canonical identifier rule:

- `supportedProfiles[]`, `chosenProfiles[]`, `timeProfileId`, and `revocationProfileId` must
  use only bytes in `[0-9a-z._/-]`
- those identifiers are ordered and deduplicated by raw byte sequence, not locale, Unicode
  normalization, or implementation-local collation

## Frame Classes

AAP traffic is divided into three major classes.

### 1. Control Frames

Used for negotiation and session-safe state changes.

- `HELLO`
- `CAPS`
- `PROPOSE`
- `COUNTER`
- `ACCEPT`
- `DECLINE`
- `CANCEL`
- `CLOSE`

Control frames are always reliable and acknowledged.

### 2. State Frames

Used for progress and coordination.

- `STATE`
- `HEARTBEAT`
- `FLOW_UPDATE`
- `BUDGET_UPDATE`
- `CHALLENGE`
- `REPAIR`

State frames are usually reliable, but may be configured as latest-only where loss is acceptable.

### 3. Bulk Frames

Used for large result and evidence payloads.

- `DELIVER`
- `PROVE`
- `CHUNK`
- `MEMORY_REF`
- `LOG_SEGMENT`

Bulk frames may be partially reliable if the contract allows it.

## Session And Stream Model

Each frame belongs to one `sessionId` and one `streamId`.

- A session represents an authenticated relationship and shared policy context.
- A resumed session may carry a new transport `sessionId` while preserving one accepted replay
  lineage.
- A stream represents a causal lane inside that session.
- Stream ordering is local, not global.
- Independent streams must not block one another.

Recommended stream types:

- `0`: session control
- `1`: capability negotiation
- `2`: contract negotiation
- `3`: execution state
- `4+`: delegated tasks and bulk transfers

Base-profile rule:

- stream `0` is reserved for `HELLO`, session close, and wire errors
- stream `1` is reserved for `CAPS`
- a peer must reject optional extensions until `CAPS` completes
- a peer must reject contract-bound frames before `HELLO` and `CAPS` complete
- every accepted session has one `sessionLineageId`, derived from the fresh bootstrap
  transcript for a new session or inherited unchanged across safe resume
- the transport-instance identity of a frame is `{sessionId, streamId, messageId}`
- the replay and dedupe identity of an admitted message is `{sessionLineageId, streamId,
messageId}`
- any header section, payload field, or audit object that references a message must carry
  both `streamId` and `messageId` unless the reference is explicitly confined to the
  current stream by schema

### Message References

Cross-stream semantics must not rely on an unscoped `MessageId`.

```text
MessageRef {
  streamId
  messageId
}
```

Rules:

- `messageId` values are unique only within one stream
- `sequence` and `AckHint` are transport-order counters, not semantic identifiers
- `MessageRef` is the canonical reference form for causality, proof binding, repair,
  challenge, and audit edges that may span streams
- any future schema that uses a bare `messageId` field must state explicitly that it is
  stream-local to the carrying frame; otherwise it is invalid in the base profile

## Base Frame Layout

Every frame starts with a fixed header followed by optional extensions and payload.

| Field          | Description                                  |
| -------------- | -------------------------------------------- |
| Magic          | 4 bytes, constant protocol marker            |
| Version        | 1 byte major, 1 byte minor                   |
| HeaderLength   | uvarint, full header length in bytes         |
| FrameLength    | uvarint, full frame length in bytes          |
| FrameType      | 2 bytes enum                                 |
| Flags          | 2 bytes bitset                               |
| SessionId      | 16 bytes opaque session identifier           |
| StreamId       | uvarint                                      |
| MessageId      | uvarint                                      |
| Sequence       | uvarint per-stream sequence                  |
| AckHint        | uvarint highest contiguous sequence observed |
| Priority       | 1 byte                                       |
| TTL            | uvarint milliseconds                         |
| HeaderBitmap   | 4 bytes presence bitmap for optional headers |
| OptionalFields | zero or more tagged header sections          |
| Payload        | typed body                                   |
| AuthTag        | variable, session protection tag             |

## Required Header Fields

### `Magic`

Protocol discriminator. Fixed constant: `0x41 0x41 0x50 0x01`.

### `Version`

Two-part version:

- major: incompatible wire change
- minor: backward-compatible field addition
- `HELLO` and pre-negotiation `CAPS` must carry fixed bootstrap version `0.0`
- any later frame must carry the negotiated `{chosenMajor, chosenMinor}`
- receiving any other `Version` before negotiation completes is a wire failure

### `HeaderLength`

Permits parsers to skip directly to payload even when they do not understand all optional fields.

### `FrameLength`

Full frame size in bytes after decryption and before reassembly at the semantic layer.

### `FrameType`

Indicates the semantic payload schema and delivery rules.

### `Flags`

Core wire flags:

- bit 0: `ackRequired`
- bit 1: `isAck`
- bit 2: `isFragment`
- bit 3: `isFinalFragment`
- bit 4: `isRetransmit`
- bit 5: `criticalExtensionsPresent`
- bit 6: `latestOnly`
- bit 7: `proofAttached`
- bit 8: `contractBound`
- bit 9: `budgetSensitive`
- bit 10: `policySensitive`
- bit 11: `reserved`
- bit 12-15: reserved

### `SessionId`

Opaque identifier generated during session establishment. Stable only within one session lifetime.

### `StreamId`

Logical channel identifier for ordering, flow control, and priority assignment.

### `MessageId`

Semantic message identifier scoped to one stream. Retransmissions reuse the same
`MessageId` on the same `streamId`.

### `Sequence`

Monotonic within a stream. Used for ack windows and gap detection.

### `AckHint`

Receiver's highest contiguous sequence observed on this stream. This supports piggybacked acknowledgments.

### `Priority`

Single-byte scheduling hint:

- `0-31`: critical control
- `32-95`: contract and state
- `96-191`: normal traffic
- `192-255`: background bulk

### `TTL`

Lifetime after which the frame is semantically stale and may be dropped.

### `HeaderBitmap`

Presence flags for optional headers defined below.

## Optional Header Sections

Optional sections are appended in ascending tag order.

### Section Encoding

Each optional header section uses one self-delimiting binary envelope:

```text
HeaderSection {
  sectionTagAndFlags
  sectionLength
  sectionValue
}
```

Rules:

- `sectionTagAndFlags` is a `uvarint`
- bit `0` of `sectionTagAndFlags` is `criticalSection`
- bits `1..` encode `sectionTag`
- `sectionLength` is a `uvarint` byte count for `sectionValue`
- `sectionValue` is exactly `sectionLength` bytes
- receivers must reject zero-length encodings for sections whose schema requires at least
  one field
- sections must appear in strictly ascending `sectionTag` order
- tags `1-32` map to `HeaderBitmap` bits `0-31`; if a tag in that range is present, its bit
  must be `1`, and if the bit is `1`, exactly one matching section must be present
- tags above `32` are encoded only through `HeaderSection` envelopes and do not consume
  `HeaderBitmap` bits
- the base profile forbids duplicate sections unless the section registry explicitly marks
  the tag repeatable
- a receiver must reject out-of-order, duplicate non-repeatable, truncated, or length-overrun
  section encodings as malformed headers
- unknown non-critical sections must be skipped using `sectionLength`
- unknown critical sections must fail the frame with wire error `102`

### Base-Profile Section Registry Rules

- tags `1-13` below are non-repeatable in the base profile
- future tags must declare repeatability, minimum encoded size, and whether omission is legal
- `criticalExtensionsPresent=1` means at least one `HeaderSection` has `criticalSection=1`
- a sender must not set `criticalExtensionsPresent=1` unless the peer acknowledged support
  for every critically offered extension tag during `CAPS`

| --- | ----------------- | -------------------------------------------- | ------------------ |
| 1   | CausalParents     | Parent message ids for causal graph          | CausalParents      |
| 2   | ContractRef       | Issuer-scoped contract identity and revision | ContractRefSection |
| 3   | BudgetRef         | Budget bucket or lease identifier            | BudgetRef          |
| 4   | PolicyRef         | Policy hash or policy manifest pointer       | PolicyRef          |
| 5   | SchemaRef         | Payload schema id and revision               | SchemaRef          |
| 6   | FragmentInfo      | Fragment group, offset, count                | FragmentInfo       |
| 7   | TraceContext      | Causal tracing and lineage ids               | TraceContext       |
| 8   | DelegationContext | Upstream delegation chain                    | DelegationContext  |
| 9   | MemoryContext     | Shared memory handles needed for decoding    | MemoryContext      |
| 10  | ProofSummary      | Compact trust metadata for attached payloads | ProofSummary       |
| 11  | GapRanges         | Sparse acknowledgment gaps                   | GapRanges          |
| 12  | CapabilityRef     | Capability-token digest and scope binding    | CapabilityRef      |

### `CausalParents`

Contains zero or more parent `MessageRef` values. This creates a DAG of semantic dependence
without forcing a total order across all streams.

Encoding rule:

- if the frame carries one or more parent references, the section encodes them as `parentRefs[]`
- if the frame has no causal parents, the section is omitted instead of emitting an empty list

### `ContractRef`

Contains:

- `issuerAgentId`
- `contractId`
- `contractRevision`
- `contractPhase`

Any frame with `contractBound` set must include this section.

This is a frame-bound reference. It is not the portable canonical-object identity type used for
signed contract artifacts outside the wire envelope.

Consistency rule:

- if the payload also carries `contractId` or `contractRevision`, those fields must byte-match
  the `ContractRef` section
- if the payload carries `issuerAgentId` for the same contract scope, it must byte-match the
  `ContractRef` section
- a mismatch is `WIRE_ERROR 114`

### `BudgetRef`

Binds the frame to a budget lease, allowing runtime enforcement without copying full budget policy into every message.

Encoding rule:

- `budgetRefId` names the active budget bucket or lease within the accepted contract or dispute policy

### `PolicyRef`

Points to the currently active policy digest or policy manifest negotiated in session setup.

Contains:

- `policyDigest`
- `policyManifestRef`
- `policyManifestInline`

Rules:

- unknown-peer base-profile traffic must bind exactly one active `TrustPolicyManifest`
- `PolicyRef.policyDigest` must equal the canonical digest of that accepted policy manifest
- a contract-bound frame whose `PolicyRef` disagrees with the session's accepted policy digest is
  invalid with `WIRE_ERROR 114`
- if `policyManifestInline` is present, its canonical digest must equal `policyDigest`
- if `policyManifestRef` is present without inline bytes, the reference must already be accepted
  in the protected session registry or be policy-pinned

### `SchemaRef`

Identifies the payload schema used to decode the frame body.

Encoding rule:

- `schemaId` and `schemaRevision` identify the concrete payload schema
- `schemaManifestRef` or `schemaManifestInline` may carry the machine-readable schema material when
  the receiver cannot rely on a previously accepted registry entry

### `FragmentInfo`

Required for fragmented bulk frames.

Contains:

- `fragmentGroupId`
- `fragmentIndex`
- `fragmentCount`
- `totalPayloadLength`

### `TraceContext`

Provides observability without changing payload semantics.

Contains:

- `traceId`
- `spanId`
- `parentSpanId`
- `lineageDepth`

### `DelegationContext`

Represents the delegation ancestry so downstream nodes know who originated, transformed, and re-assigned the work.

Encoding rule:

- `lineagePath[]` is the ordered actor path from originator to current sender
- `delegationDepth` is the effective number of delegation hops represented by `lineagePath[]`
- `rootCapabilityTokenDigest` binds the chain back to the root capability when delegated authority
  is the basis for execution

### `MemoryContext`

Contains one or more `MemoryHandle` references needed to interpret the payload.

Encoding rule:

- each `MemoryHandle` identifies a shared segment by digest and the exact byte range needed for decoding

### `ProofSummary`

Provides compact trust hints before the receiver parses the proof body.

Contains:

- `trustLevel`
- `evidenceKindBitmap`
- `proofDigest`
- `verifierHint`

### `GapRanges`

Present only on ack-bearing frames.

Contains:

- `ackBaseSequence`
- `missingRanges[]`

`missingRanges[]` expresses gaps above `AckHint` so a sender can retransmit precisely
without inferring loss from timeout alone.

### `CapabilityRef`

Binds a permission-sensitive frame to the capability material that authorized it.

Contains:

- `capabilityTokenDigest`
- `tokenClass`
- `grantedScopeDigest`
- `delegationDepth`

### `FinalityContext`

Connects settlement or close traffic to externally checkable finality evidence.

Contains:

- `finalityMode`
- `finalityDigest`
- `witnessSetDigest`
- `observationDigest`

## Payload Encoding

Payloads use schema-bound compact objects.

### Envelope Rule

Every payload begins with:

- `schemaLocalRevision`
- `bodyLength`
- `body`

Envelope encoding rule:

- `schemaLocalRevision` is a canonical `uvarint`
- `bodyLength` is a canonical `uvarint`
- `body` is exactly `bodyLength` bytes of the selected payload encoding

### Encoding Profile

- Small structured bodies: canonical tagged binary object encoding `aap-tagbin-v1`
- Logs: typed segment list
- Embeddings and vectors: packed numeric arrays
- Large artifacts: chunk stream with out-of-band assembly metadata

### Canonical Encoding Rules

Base-profile structured payloads must obey all of the following:

- field tags are unique positive integers
- fields are encoded in ascending tag order
- omitted optional fields are not serialized
- default values are never serialized unless the field is explicitly marked present
- the base profile rejects unknown payload field tags unless a later published profile explicitly
  marks that payload family as extension-preserving and defines the required decode and re-encode
  behavior
- if a later published profile permits preservation of unknown non-critical fields across
  decode/re-encode, those fields must retain their original field-entry bytes inside the payload
  body even though the surrounding session header and auth tag are recomputed
- digests and signatures are always computed over the canonical encoded body

`aap-tagbin-v1` object rule:

```text
TagBinObject := FieldEntry*
FieldEntry {
  fieldTag
  valueLength
  valueBytes
}
```

Canonical object rules:

- `fieldTag` is a canonical `uvarint`
- `valueLength` is a canonical `uvarint`
- `valueBytes` is exactly `valueLength` bytes
- scalar fixed-width integers encode in big-endian byte order with no alternate widths
- `uvarint` uses unsigned LEB128 shortest form; zero encodes as one byte `0x00`
- `bytes` and `text` use raw field bytes without an inner length prefix because the outer field
  envelope already carries `valueLength`
- base-profile `digest` uses raw digest bytes without an inner length prefix; unless a specific
  field table or later negotiated profile says otherwise, the digest algorithm is SHA-256 and the
  encoded value is exactly the 32 digest bytes rather than the textual `sha256:...` projection used
  in Markdown
- nested objects encode as canonical `TagBinObject` bytes inside `valueBytes`
- `manifestRef`, `messageRef`, `range`, and any future structured composite type encode as
  nested canonical objects using their declared field tables
- `list<T>` encodes as one nested `TagBinObject` carried inside the parent field entry
- each list element is encoded as a repeated child field with `fieldTag = 1`, preserving the
  sender's canonical element order
- the child field's `valueBytes` is the canonical encoding of exactly one element
- scalar fixed-width element types use their canonical scalar bytes as the child `valueBytes`
- `text`, `bytes`, `digest`, and composite element types use the same canonical bytes they would
  use as a standalone field value, delimited by the child field's `valueLength`
- a decoder must reject any list encoding that cannot be consumed as an ordered sequence of whole
  child entries with `fieldTag = 1`
- an empty object or empty list is encoded by omission unless the schema explicitly marks
  presence as semantically significant

Handshake-specific rule:

- a base-profile peer must reject `HELLO` or `CAPS` if a mandatory digest is advertised
  without a corresponding reference and the referenced artifact is not already pre-pinned
  by local policy

`aap-tagbin-v1` field-type registry:

- `u8`, `u16`, `u32`, `u64`: fixed-width unsigned integers
- `uvarint`: variable-width unsigned integer
- `bool`: single-byte `0` or `1`
- `bytes`: opaque bytes carried directly in `valueBytes`
- `text`: UTF-8 bytes carried directly in `valueBytes`
- `digest`: canonical binary digest bytes carried directly in `valueBytes`; the base-profile default
  algorithm is SHA-256 unless the field table or negotiated profile declares another algorithm
- `manifestRef`: canonical `ManifestRef` object encoding
- `messageRef`: canonical `MessageRef` object encoding
- `list<T>`: one enclosing field whose value is a canonical child `TagBinObject` made only of
  repeated child entries with `fieldTag = 1`; the ordered concatenation is the concatenation of
  those child field encodings, not raw scalar bytes
- `range`: `{minInclusive,maxInclusive}` encoded as two child `uvarint` fields

  the accepted policy digest for audit
- if `acceptedTrustPolicyInline` is present, its canonical digest must equal
  `acceptedTrustPolicyDigest`

## Reliability Profiles

AAP does not force one delivery rule for all traffic.

### Reliable

- Retransmit until acked or TTL expired.
- Preserve per-stream ordering.
- Used for control and contractual state.
- every reliable frame consumes one `sequence` value
- pure ack frames also consume one `sequence` value so loss detection remains monotonic
- ack-only frames must never require acknowledgment in return
- a sender must keep the canonical protected bytes of each unacked reliable frame until it
  is acknowledged, declared stale, or the session is closed

### Latest-Only

- Receiver accepts newest frame and may drop older queued frames with the same semantic key.
- Used for heartbeat and high-rate progress telemetry.
- latest-only frames still carry `sequence`, but gaps caused only by dropped older
  latest-only frames must not trigger retransmission requests

`semanticKey` must be a stable tuple of:

- `contractId`
- payload `phase`
- progress lane identifier defined by schema

If a schema does not define a semantic key, latest-only delivery is forbidden.

### Partial-Reliable

- Sender retransmits up to the deadline or retry budget.
- Useful for bulk output where stale chunks are not worth full recovery effort.
- partial-reliable frames must declare retry budget and fragment policy through contract or
  schema profile before use

Reliability is selected by frame class and may be narrowed by contract.

## Fragmentation And Reassembly

Frames larger than path-safe datagram size are fragmented at the AAP frame layer after
canonical payload assembly and before transport packetization.

Rules:

- Fragments share `MessageId`.
- `FragmentInfo` must be present on every fragment.
- Control frames must not be fragmented unless unavoidable.
- Receivers may reject incomplete fragmented control messages.
- Bulk payloads can be streamed before full reassembly only if schema marks them as progressive.
- each fragment is a complete protected AAP frame with its own auth tag
- all fragments of one message must use the same `streamId`, `MessageId`, `ContractRef` when
  present, and canonical protected payload digest
- on reliable or partial-reliable streams, each fragment consumes its own `sequence`; only
  retransmission may reuse that fragment `sequence`
- `fragmentIndex` is zero-based and `fragmentCount` is the total number of fragments
- `fragmentCount=0` is invalid
- a receiver must reject duplicate fragment indexes carrying different protected bytes as
  equivocation or corruption
- wire receipt and retransmission operate at fragment-frame granularity, while semantic
  acceptance operates only after all mandatory fragments for the message validate and reassemble
- incomplete fragment groups for control or state frames must expire no later than the
  smaller of `TTL` and `30000ms`
- incomplete fragment groups for bulk frames must expire no later than the smaller of `TTL`
  and the contract's declared delivery deadline
- a base-profile receiver must bound concurrent fragment groups per peer and per contract;
  exceeding policy must emit `WIRE_ERROR 106` and discard the newest excess groups first

## Flow Control

Flow control operates at three scopes:

1. Session scope: total bytes in flight
2. Stream scope: per-stream window
3. Contract scope: work budget and result budget

`FLOW_UPDATE` frames advertise:

- receive window
- target pacing rate
- congestion signal
- preferred priority shedding threshold

Units:

- receive window: bytes
- target pacing rate: bytes per second
- congestion signal: integer in range `0-1000`
- preferred priority shedding threshold: highest priority value to begin shedding

## Deadlines

Each frame carries `TTL`, but contracts may also provide:

- execution deadline
- proof deadline
- settlement deadline

If frame `TTL` exceeds its parent contract deadline, the receiver clamps it to the contract deadline.

Base-profile rule:

- `TTL=0` is invalid
- receipt after `TTL` expiry must be treated as stale
- retransmission is forbidden after local `TTL` expiry even if retry budget remains
- absolute contract deadlines must be evaluated using the portable skew rule, not local
  logical time alone
- a stale frame may still be auditable, but it must not mutate semantic state
- deadline comparison is inclusive at the exact boundary; a frame observed at
  `localObservedAtUnixMs == effectiveDeadlineUnixMs` is still timely
- when multiple protected frames become wire-admissible in the same local receive batch, the
  batch must be normalized into ascending `{sequence, streamId, messageId}` order before any
  semantic precedence rules from `02.CONTRACT.MODEL.md` are applied

| Key                        | Value                                             |
| -------------------------- | ------------------------------------------------- |
| `deadlineBoundaryRule`     | `inclusive`                                       |
| `stalePredicate`           | `localObservedAtUnixMs > effectiveDeadlineUnixMs` |
| `sameBatchOrderingFields`  | `sequence, streamId, messageId`                   |
| `sameBatchOrderingPurpose` | `semanticAdmissionDeterminism`                    |

## Acknowledgment Model

Acknowledgment is selective.

- `AckHint` covers contiguous success.
- Optional gap ranges may be attached through a tagged extension.
- Receivers may acknowledge semantic completion separately from wire receipt.

Ack-bearing frames use `isAck=1` and a zero-length payload unless another payload is
coalesced intentionally. Ack processing is idempotent.

Normative ack rules:

- `AckHint` acknowledges all reliable `sequence` values less than or equal to `AckHint`
  on the same stream except those explicitly listed in `GapRanges`
- `GapRanges` is relative to `AckHint` and must be normalized into non-overlapping,
  ascending ranges
- an ack-bearing frame must never acknowledge a sequence that the receiver has not yet
  validated at the protected-byte level
- a sender must treat an acknowledged reliable frame as final wire receipt and stop
  retransmitting it
- a sender may retransmit an unacked reliable frame only with identical canonical protected
  bytes and the same `{streamId, MessageId, sequence}`
- a receiver that observes a sequence gap on a reliable stream should wait at least one
  local reordering interval before emitting gap-based retransmission pressure
- base-profile default timers are:
  - ack delay ceiling: `25ms` for control streams, `100ms` otherwise
  - initial retransmit timeout: `500ms`
  - retransmit backoff: multiplicative factor `2.0`
  - maximum retransmit timeout: `8000ms`
- peers may negotiate stricter timers, but a sender must not assume lower values until both
  sides advertise the same timer profile

### Wire Receipt Versus Semantic Finality

- wire acknowledgment proves protected receipt only
- semantic acceptance requires payload validation against contract, schema, and policy
- trust elevation requires proof validation and any required verifier outcomes
- settlement finality requires the contract's declared signatures, observations, or
  witness attestations in addition to receipt of `SETTLE` or `CLOSE`
- implementations must expose these checkpoints separately in audit and observability

Duplicate handling:

- retransmissions must reuse `MessageId`
- a receiver must deduplicate admitted traffic by `{sessionLineageId, streamId, MessageId}`
- before `sessionLineageId` is established, duplicate suppression may use the transport
  `sessionId` only for the bootstrap exchange
- duplicate control frames must not re-apply side effects
- duplicate bulk frames may refresh liveness timers but must not create new contract state
- if the same `{sessionLineageId, streamId, MessageId}` arrives with different canonical protected
  bytes, the receiver must treat it as equivocation or corruption and emit `WIRE_ERROR 100`
  or `108` according to local replay-window context
- dedupe state for reliable control frames must persist until session close and, for
  contract-bound terminal frames, until the contract reaches `Closed`
- a receiver must reject any contract-bound frame whose `contractRevision` is lower than the
  highest accepted non-terminal revision for that contract unless the frame is a duplicate of
  already accepted canonical bytes
- after contract `Closed`, non-duplicate contract-bound frames for that contract must be
  rejected as stale or replayed traffic

Replay-window rule:

- base-profile receivers must retain dedupe state for at least the larger of `2 *` the
  maximum retransmit timeout and the remaining contract TTL for reliable control frames
- session resumption must restore or safely invalidate this dedupe state before accepting
  resumed contract traffic

This distinction matters because a `DELIVER` frame can arrive intact while still failing semantic validation.

## Session Resumption And Replay Continuity

Resumption is valid only if it preserves replay and transcript safety.

Rules:

- `resumeTokenPresent` advertises an intent to resume, not a right to resume
- `resumeTokenBindingDigest` binds the presented resume token to the prior accepted bootstrap
  transcript and the prior session replay window state
- unknown-peer resumption is valid only if the resumed `HELLO` and `CAPS` still converge on the
  same `acceptedTrustPolicyDigest` used for the resumed lineage
- successful resumption must preserve the prior `sessionLineageId`; failed preservation
  requires rejecting resumption and starting a fresh lineage instead
- a responder may accept resumption only if the resume token is policy-valid, freshness-valid,
  and bound to the same peer identity and negotiated major version family
- accepted resumption must restore dedupe state for unexpired reliable frames or reject
  resumption and require a fresh session instead
- if replay continuity cannot be restored safely, the peer must refuse resumption rather than
  silently widening the replay window
- a policy change that produces a different accepted trust policy digest must emit
  `ResumeRefusal.refusalReasonCode = trustPolicyMismatch` and require a fresh session rather than
  carrying resumed contract traffic under the prior lineage
- a resumed session must surface a new `sessionId`, but its audit trail must link the new
  bootstrap transcript to the prior one through the accepted resume binding
- contract traffic must remain blocked until resumed `HELLO` and `CAPS` complete under the
  new protected session
- any session-bound authority artifact used after resume must carry both stable
  `sessionLineageId` and concrete `sessionId` binding semantics; lineage match alone is
  insufficient to authorize reuse under a new transport session
- if a session-bound artifact's `sessionId` does not match the active resumed transport session,
  the peer must require either a canonical rebinding artifact or a freshly issued replacement
  before granting the artifact's authority
- accepted resumption must be auditable through `ResumeAcceptance`
- refused resumption must be auditable through `ResumeRefusal`
- after a successful resume, a duplicate of any previously admitted reliable control frame must
  be rejected or treated as a byte-identical no-op according to the restored replay window; a
  divergent duplicate must emit `WIRE_ERROR 108`

## Version Negotiation

Version negotiation happens during session establishment.

Rules:

- Bootstrap establishes the cryptographic envelope only.
- `HELLO` and the initial `CAPS` exchange always use bootstrap wire version `0.0`; that value
  is a negotiation container, not an advertised supported production version
- `HELLO` advertises `supportedMajorVersions[]` and one `supportedMinorRanges[]` entry for
  each advertised major at the same index.
- the chosen major version is the numerically highest value present in both peers'
  `supportedMajorVersions[]`.
- if there is no common major version, negotiation fails.
- for the chosen major version, the chosen minor is the highest value in the intersection of
  both peers' indexed minor ranges for that major.
- `CAPS` must carry `chosenMajor` and `chosenMinor` explicitly; contract traffic before both
  peers emit matching chosen values is invalid.
- `chosenProfiles[]` is the stable intersection of both peers' supported profile identifiers,
  sorted in ascending canonical identifier order.
- `offeredCriticalExtensions[]` is the subset of `supportedExtensions[]` that the sender may emit
  with `criticalSection=1` in the current session; if omitted, treat it as the empty set.
- `acceptedCriticalExtensions[]` is the deterministic intersection of both peers'
  `offeredCriticalExtensions[]`; use of any other critical extension is invalid.
- if peers emit mismatched `chosenMajor`, `chosenMinor`, `chosenProfiles[]`, or
  `offeredCriticalExtensions[]`, or `acceptedCriticalExtensions[]`, the session must fail rather
  than proceed with local interpretation.
- manifest-backed capability, schema, and unit references must resolve or already be
  policy-pinned before profile negotiation is considered complete
- time-profile and revocation-profile references must resolve or already be policy-pinned
  before contract traffic is permitted
- if the negotiated profile requires identity or bootstrap-artifact revocation freshness,
  that freshness check must complete before `CAPS` succeeds
- ties in profile or extension order must be broken only by canonical identifier ordering,
  never by sender preference text or implementation-local defaults
- once negotiation succeeds, every subsequent frame in the session must carry the chosen wire
  version; reverting to `0.0` outside a fresh or resumed handshake is invalid

Deterministic negotiation algorithm:

1. compute the intersection of `supportedMajorVersions[]`
2. choose the numerically highest common major
3. resolve each peer's minor range for that chosen major by position from
   `supportedMinorRanges[]`
4. intersect those ranges and choose the highest common minor
5. compute `chosenProfiles[]` as the sorted set intersection of `supportedProfiles[]`
6. validate that each peer's `offeredCriticalExtensions[]` is an ascending unique subset of that
   peer's `supportedExtensions[]`
7. compute `acceptedCriticalExtensions[]` as the sorted set intersection of both peers'
   `offeredCriticalExtensions[]`
8. compute effective size and recursion limits by pairwise minimum
9. fail the session if either peer emits any different result for steps `2-8`

If any advertised major lacks a corresponding minor-range entry, `HELLO` is malformed and
must fail with `WIRE_ERROR 100`.

## Conformance Corpus

Independent implementations should ship and consume a shared wire corpus.

Required golden vectors:

- one canonical `HELLO` / `CAPS` transcript for a clean unknown-peer join
- one canonical resume transcript that succeeds without widening replay state
- one authenticated bootstrap transcript that includes transcript-binding, replay-window, and
  resume-token derivation outputs from the public secure-session companion profile
- one canonical bootstrap transcript that includes trust-policy negotiation and the resulting
  `BootstrapAcceptance.acceptedTrustPolicyDigest`
- one canonical `PROPOSE -> ACCEPT -> STATE(ready) -> STATE(executing)` lifecycle trace
- one canonical `DELIVER -> PROVE -> SETTLE -> CLOSE` terminal trace with witness-backed
  finality
- one canonical latest-only `STATE` stream that demonstrates stable `semanticKey` handling
- one canonical fragmented bulk payload with successful reassembly
- one canonical bootstrap `Version=0.0` handshake that transitions to a negotiated production
  version without ambiguity

Required negative vectors:

- malformed `HELLO` with mismatched `supportedMajorVersions[]` and `supportedMinorRanges[]`
- malformed bootstrap frame using a non-`0.0` pre-negotiation `Version`
- `HELLO` or resumed `CAPS` carrying a trust policy digest that does not match the accepted
  session policy
- `CAPS` with mismatched chosen version, profiles, or negotiated limits
- duplicate admitted `{sessionLineageId, streamId, MessageId}` carrying different protected
  bytes
- contract-bound frame whose header `ContractRef` disagrees with repeated payload identity
- fragmented frame with conflicting fragment bytes or impossible `fragmentCount`
- ack frame with overlapping or descending gap ranges
- contract-bound frame before handshake completion
- `CLOSE` without the finality evidence required by the accepted contract

Each vector must include:

- canonical protected frame bytes
- companion-profile bootstrap inputs and authenticated transcript outputs when the case
  exercises session bootstrap or resumption
- expected parse tree
- expected semantic outcome
- expected wire error code when the case is negative

## Implementation Notes

- Parsers should pre-validate length fields before allocation.
- Stream schedulers should prioritize control frames by default.
- Unknown non-critical fields must be preserved if the frame is forwarded.
- Message ids should be large enough to avoid wraparound within session lifetime.
- Proof-bearing frames should expose `ProofSummary` so trust routing can happen early.
- Base-profile implementations should ship golden frame captures for every mandatory schema.
