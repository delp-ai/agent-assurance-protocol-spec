# AAP Conformance Safe Resume

This artifact defines canonical object-form resumption behavior for the revised `AAP`
base profile.

It focuses on one requirement only:

- resumption is admissible only when transcript binding and replay continuity are preserved

Scenario note:

- `ResumeAcceptance` and `ResumeRefusal` below are the normative canonical artifacts
- `ResumeCaseOutcome` is an informative conformance summary used only to state scenario expectations;
  it is not a peer-visible wire object and it is not part of the base-profile canonical object set

## Participants

- initiator: `agent.alpha`
- responder: `agent.beta`

## Duplicate-After-Resume Rejection

### Precondition

- `ResumeAcceptance.canonicalDigest = sha256:resume_accept_0002` was already accepted
- the restored replay window already retains admitted message identity
  `{sessionLineageId: lineage.boot_0001, streamId: 3, messageId: 4}`

### Input

```text
Frame {
  Version: 1.0
  StreamId: 3
  MessageId: 4
  FrameType: CHALLENGE
  ContractRef {
    issuerAgentId: agent.alpha
    contractId: contract.exec.0001
    contractRevision: 1
    contractPhase: challenged
  }
  PayloadDigest: sha256:challenge_exec_DIVERGENT
}
```

Expected outcome:

```text
ResumeReplayOutcome {
  accepted: false
  emittedWireError: 108
  failureReason: divergentDuplicateAfterResume
  replayWindowWidened: false
}
```

## Required Rule

Resumption is valid only if:

- the peer identity still matches
- the resume token is freshness-valid
- the resumed transcript binds to the same accepted lineage
- the resumed handshake converges on the same accepted trust policy digest
- replay-window state is restored exactly enough to reject divergent duplicates
- a successful resume produces one canonical `ResumeAcceptance`
- a refused resume produces one canonical `ResumeRefusal`

If any of those checks fail, the implementation must refuse resumption and require a fresh
session rather than widening replay acceptance.
