# AAP Canonical Object Encoding

## Purpose

This document defines the canonical encoding profile for signed or digested base-profile
objects that are not fully described by frame payload tables alone.

For unknown-peer interoperability, these objects are part of the public protocol surface.

## Canonical Encoding Profile

All objects in this document use `aap-tagbin-v1`.

Base rules:

- fields are encoded in ascending tag order
- absent optional fields are omitted
- repeated list fields preserve the sender's canonical order exactly as required by the carrying
  object definition; if a field is set-like, ascending unique, or bytewise sorted, that rule must
  be stated explicitly by the field table notes or surrounding normative text before encoding
- every `list<T>` field encodes as one enclosing field whose value is a canonical child
  `TagBinObject` consisting only of repeated child entries with `fieldTag = 1`
- each list child entry carries exactly one canonical element encoding in its `valueBytes`
- nested objects are encoded as canonical child objects inside the field value
- booleans encode as `00` for false and `01` for true
- text fields use visible ASCII unless the referenced object profile explicitly permits UTF-8
- integers use the same base types defined in `01.WIRE.FORMAT.md`
- `bytes`, `text`, and `digest` use the same raw `valueBytes` rules defined in
  `01.WIRE.FORMAT.md`; they do not carry any inner length prefix
- unknown non-critical fields must be ignored only when the carrying object profile explicitly
  permits extension and publishes the preservation rules needed for digest-stable re-encoding;
  otherwise unknown fields are invalid in the base profile

## Digest And Signature Scope

Unless a specific object says otherwise:

- `canonicalDigest` or `attestationDigest` is computed over the object's canonical bytes with
  the digest field omitted and any trailing `signature` or `signatureSet[]` fields omitted
- unless a specific object or negotiated profile says otherwise, base-profile digest fields and
  signature inputs use SHA-256 over those canonical bytes
- `signature` covers the computed canonical digest
- `signatureSet[]` entries must each cover the same canonical digest
- peers must reject objects whose recomputed digest does not match the carried digest

### `FrameContractRef`

Use the same field tags as `ContractRefSection` in `01.WIRE.FORMAT.md`.

Canonical-object compatibility rule:

- unless a field table explicitly names `revisionRef` or `frameContractRef`, a field typed as
  `contractRef` means `PortableContractRef`
- canonical objects must not encode `contractPhase` inside `contractRef`; phase belongs only in
  `FrameContractRef` or in an explicit sibling field such as `fromPhase`, `toPhase`, or
  `priorResultPhase`
- when a canonical object also carries `contractRevision`, that field is the only revision source
  for canonical digest computation
- canonical objects must not infer revision from `contractRef` alone; a revision-sensitive object
  must carry an explicit sibling revision field

### `ManifestRef`

Use the same field tags as `ManifestRef` in `01.WIRE.FORMAT.md`.

Reference typing rule:

- any field typed as `manifestRef` in this document must resolve through the manifest-resolution
  rules in `01.WIRE.FORMAT.md`; interoperable implementations must not replace it with
  implementation-local locator semantics
- any field that remains `text` even though its name ends in `Ref` is a canonical symbolic
  identifier, not a free-form human locator; the profile that defines it must still make
  resolution deterministic
- if a later profile wants a portable dereferenceable artifact instead of a symbolic registry key,
  it must promote that field to `manifestRef` or another explicitly typed composite reference

## Conformance Rule

Any object listed in this document is part of the base-profile public conformance surface
whenever:

- it is referenced by digest or signature in a mandatory protocol path
- it gates trust upgrade, settlement, dispute, resumption, or irreversible close
- it is required for unknown-peer interoperability claims

Unknown-peer `AAP Open Core` claims must therefore ship:

- canonical object bytes or canonical object forms aligned to these field tags
- expected digests for positive fixtures
- deterministic rejection reasons for malformed, stale, or equivocated fixtures
