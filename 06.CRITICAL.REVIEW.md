# AAP Critical Review

## Purpose

This document is presenting a third-party autonomous-agent review of the current `AAP`
document set.

The review is intentionally written from a machine-only standpoint:

- the reviewer is an external agent
- the reviewer is also the effective user of the protocol
- no human approval, interpretation, escalation, or override is assumed at any step

## Review Standard

`AAP` is judged by whether an unknown external agent could:

- join without vendor-local setup
- resolve every required bootstrap, contract, and trust artifact without human help
- compute the same canonical digests and parse results as another implementation
- survive replay, challenge, timeout, bridge, and finality races without ambiguous behavior
- distinguish receipt, semantic acceptance, verification, and final closure
- terminate disputes and settlement through machine-checkable outcomes only

## Executive Verdict

The specification set is supporting implementation for:

- controlled federation inside one trust domain
- pre-configured bilateral or multi-cell deployments
- bounded public unknown-peer `AAP Open Core` interoperability, but only for the exact maintained
  surface published in this repository

The repository is operating as more than a prose protocol draft. It is publishing:

- normative wire, contract, trust, and canonical-object documents
- generated SSOT for schemas, registries, objects, byte fixtures, and bindings
- positive and negative conformance artifacts
- self-contained unknown-peer bootstrap byte fixtures
- transcript-level replay fixtures and mutation-tested validators
- machine-readable claim-surface bindings, transcript dependency metadata, and a subprocess replay
  harness for the maintained public bundle

## Current Strengths

- `01.WIRE.FORMAT.md` is separating protected receipt from semantic acceptance and finality instead of
  collapsing them into one event.
- `02.CONTRACT.MODEL.md` is making authority, budgets, dispute policy, settlement, and transition rules
  explicit enough for independent implementation.
- `03.TRUST.MODEL.md` is treating trust-domain independence, revocation freshness, bridge bindings, and
  finality as protocol state rather than implementation-local policy.
- `04.CANONICAL.OBJECTS.md` is giving one interoperable canonical encoding model for signed and digested
  non-frame objects.
- `conformances/` is publishing a maintained gating corpus with semantic traces, exact frame bytes,
  canonical object bytes, transcript fixtures, and negative cases.
- The maintained corpus is including a self-contained unknown-peer bootstrap fixture set rather than
  relying only on declared bootstrap context.

## Current Claim Boundary

Public unknown-peer interoperability may be claimed only when an implementation reproduces the
published mandatory surface exactly:

- the secure bootstrap and safe-resume companion-profile rules
- canonical encoding for the base-profile signed and digested objects it emits or verifies
- machine-determined negotiation, dispute, timeout, bridge, and finality behavior published in the
  maintained corpus
- the self-contained bootstrap fixture set plus the maintained resume, terminal-close, and negative
  fixtures in `conformances/`, each with the published dependency class and prerequisite context it
  declares
- deterministic rejection of reserved extension frame codes and unpublished extension surfaces

The current claim surface is also projecting into `specs/bindings.json` so an outside agent can
consume the maintained required artifact set without inferring it from prose alone.

The maintained byte transcript set is also declaring dependency classes and prerequisite contexts in
`specs/bytes.json`, so replay scope is staying machine-visible rather than narrative-only.

The repository does not justify broader claims for:

- unpublished extension profiles
- implementation-local registries or manifest conventions
- operator-only bootstrap shortcuts
- behavior inferred from guidance rather than from the maintained authored and validated surface

## Current Risks

The main remaining risks are implementation and ecosystem risks, not missing protocol law.

- Implementations still have to preserve the companion-profile transcript scope exactly when they
  wrap and unwrap the standardized protected AAP bytes.
- Deployments that narrow trust roots, discovery inputs, bridge behavior, or manifest sources must
  surface those restrictions explicitly as deployment-profile limits.
- Cross-vendor interoperability still depends on multiple independent implementations reproducing the
  maintained corpus, not only on one repository-local validator passing.
- The maintained byte corpus is strong, but it is still a maintained subset of total semantics; new
  mandatory public claims should continue to be accompanied by promoted byte fixtures and validator
  coverage.

## Readiness Split

### Controlled Federation

Implementation is proceeding now.

Recommended scope:

- secure session manager with a pinned bootstrap profile
- frame codec and stream scheduler
- manifest resolver
- contract state machine
- challenge and repair handling
- append-only audit chain
- capability enforcement
- metering and settlement reduction
- trust and finality validation against pinned schemas

### Unknown-Peer Open Federation

Implementation is staying viable for the currently published `AAP Open Core` mandatory surface.

Required posture:

- implement the maintained self-contained bootstrap fixtures and safe-resume rules exactly
- implement the published canonical object schemas exactly
- reproduce the maintained corpus outcomes without human interpretation
- keep unpublished or provisional extension behavior disabled or deterministically rejected
- avoid claiming interoperability for any wider surface than the repository currently publishes and
  validates

## Bottom Line

`AAP` is operating as a serious autonomous-agent protocol design.

From a purely agent-only standpoint:

- it is supporting build-out now for controlled federation
- it is supporting bounded unknown-peer public interoperability claims for the maintained
  `AAP Open Core` surface while bootstrap closure, canonical objects, conformance fixtures, claim
  manifests, and transcript dependency metadata are all staying published in machine-checkable form
- the right next step is more independent implementation and cross-vendor replay testing, not
  another round of high-level protocol redesign
