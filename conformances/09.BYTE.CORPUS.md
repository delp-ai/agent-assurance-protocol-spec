# AAP Conformance Byte Corpus

This artifact defines the byte-level format for `AAP` conformance vectors and provides the
canonical byte fixtures that back the transcript and semantic gating corpus.

## Corpus Convention

These vectors use a conformance-only protection convention:

- bytes are post-decryption AAP frame bytes
- `AuthTag` length is fixed to `0` for these vectors only
- therefore `FrameLength` in these examples covers bytes from `Magic` through the end of the
  payload

This convention does not replace the real secure-session layer. It standardizes the exact AAP
protected bytes that independent implementations must agree on after authenticated session
protection is removed and before any transport-specific re-protection is applied.

Scope rule:

- this byte corpus is a maintained golden subset of the full AAP public surface, not a promise
  that every mandatory semantic branch already has its own standalone byte vector in this file
- implementations claiming unknown-peer interoperability must pair these byte vectors with the
  semantic corpus, transcript corpus, and canonical object tables rather than treating this file
  alone as a complete protocol oracle

## Fixed Test Inputs

Unless a vector says otherwise, use:

```text
sessionId = 00 01 02 03 04 05 06 07 08 09 0A 0B 0C 0D 0E 0F
ttl = 0x64
criticalControlPriority = 0x10
contractPriority = 0x20
schemaLocalRevision = 0x01
```

Encoding reminders:

- fixed-width integers are unsigned big-endian
- `uvarint` is unsigned LEB128 shortest form
- `aap-tagbin-v1` field entries encode as `fieldTag valueLength valueBytes`
- optional header sections use `sectionTagAndFlags sectionLength sectionValue`

## Digest Alias Convention

The byte corpus uses compact conformance-only digest aliases inside exact frame and object bytes.

Rules:

- a one-byte digest token in this file is a corpus alias, not a real production hash output
- independent parsers must validate the byte width exactly as published in this corpus instead of
  inferring a longer digest length from the semantic alias name
- aliases are vector-local by default; reusing the same raw alias byte in two different vectors does
  not, by itself, assert that the underlying semantic digest is the same
- cross-vector digest identity exists only for aliases listed in the explicit shared-alias table
  below; all other identical raw alias bytes must be treated as unrelated corpus-local placeholders
